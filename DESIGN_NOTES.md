## 🎯 趣味性大修 ROADMAP（2026-05 诊断后规划）

> 来源：2000 局 sim + 全卡内容审计 + 玩家"无聊"反馈。
> 三大病灶：① 牌库膨胀但触达率低 ② 攻击牌占比过高 ③ 决策密度低 / 缺 burst combo timing

### P0：必做四件（~100 分钟工作量）

#### P0-1：起手牌库精简 40 → 25 张
- 起手 21 攻击 → 12（每色 3 张）
- 起手 6 技能 → 4（保留 sk_focus 摸牌 + sk_dye 染色 + 2 个有意思的）
- 起手 6 道具 → 2（it_heal ×1 + it_bomb ×1）
- 起手 7 装备保留
- **预期触达率：50% → 70%**

#### P0-2：奖励池清理
- `atk_X` **移出** REWARD_CARD_POOL_BASE（玩家抽到攻击牌体验崩）
- AOE 池整合进主池但权重低（或保留独立 F3+ 触发）
- 奖励池 73 → ~55 张

#### P0-3：奖励抽保底机制
- 3 张候选改为 **固定 1 common + 1 rare + 1 super_rare/epic**
- 玩家每场战斗必拿稀有以上
- forceEquipment 装备保底机制保留

#### P0-4：数值通胀修复（混合方案，不一刀切百分比）

| 卡类型 | 缩放方式 | 理由 |
|---|---|---|
| 单体直伤 | `base + floor × N` | 缩放慢于敌人 HP（1.18^f），法术做辅助 |
| 群伤大招 | maxHP %（已用） | 避免一卡清场 |
| Debuff 层数 | `base + floor / N` 缓增 | 状态机制保留 |
| 单次效果太弱 | 延长 duration | freeze 类 |
| 机制收益不可控 | 重做机制 | recolor / silence |
| 资源置换 | 降代价 | blast 类 |

**具体 9 张需要修的卡：**

| 卡 | 当前 | 改为 |
|---|---|---|
| it_bomb 炸弹 | 15 直伤 | `8 + floor × 2`（F1=10 / F12=32） |
| sk_dbl_pummel 双重打击 | 4 直伤 + 易伤 | `(4 + floor)` 直伤 + 易伤 2 回合 |
| sk_pierce_strike 穿甲斩 | 下次 +3 pierce | 下次 +`(3 + floor/3)` pierce |
| sk_blast 爆裂术 | 自损 10% / 30% 当前 HP | 自损 5% / 30% 当前 HP（降成本） |
| sk_curse_vortex 诅咒漩涡 | 全体 +2 中毒 | 全体 +`(2 + floor/3)` 中毒 + 易伤 2 回合 |
| sk_freeze 冰冻 | -50% 伤害 1 回合 | -50% 伤害 **2 回合** |
| sk_silence 沉默 | 跳过 1 buff | **"下回合敌人技能/buff 跳过 + 攻击保留"**（机制不变但保留攻击让其更通用） |
| sk_recolor 变色 | 目标花色随机 | **"目标花色变为玩家激活花色 4 回合"**（保护亲和度） |
| it_purify 驱毒剂 | 清自身 dot | 清自身 dot + **摸 1 张** |

### P1：进阶（待 P0 验证后再做）

#### P1-1：Power 卡分类
- 类似 STS Power：战斗内出，永久 buff 本场
- 创造 setup / burst 节奏（开场 stack power → 中后段爆发）
- 待用户提供 4-6 张 Power 卡方向（如"♠ 磨刀 +1 攻击/出攻击"）

#### P1-2：Status 合并清理
- `dyed_♠♦♥♣` + `chanted_♠♦♥♣` 8 个 → 合并 1 个 status 用 stack 编码花色
- 6 个附魔触发状态（phantom_charge / phalanx_dr / swift_dodge_temp / enc_runic_immune / enc_dot_immune / warblood_perm_atk）改为**隐藏 buff**（不在状态栏，只在卡描述）

### P2：不推荐（先不做）

- **能量系统**（每回合 N 能量，每张卡 1-3 cost）— 核改造，会冲击花色亲和度系统，复杂度爆炸
- **卡升级 +形态**（DESIGN_NOTES Slot D 挂着的）— 工作量大，先不动

### P3：附魔 5 段式重复增强（用户已确认方向，等实施）

- 一把武器只能挂一种附魔，但**可反复进铁匠铺再施加**升级
- 每次施加 +1 级（cap 5 级），按原配方收费
- 例如 e_brawler 强袭：L1 攻击 +5% → L2 +10% → … → L5 +25%

## 🅒 闪避 / 穿甲系统补完（待实现）

### 当前问题
1. **闪避：**
   - 意念甲（10/20/30/40% 完全闪避）和特性 p_dodge（每张 +3% 完全闪避）独立 roll，玩家算不清最终概率。
   - "闪避姿态"（技能：受击减半）和"完全闪避"是两个完全不同的概念，名字撞了。
   - 战斗 UI 不显示当前闪避概率。
   - 触发时只有日志一行，毫无爽感。

2. **穿甲：**
   - 全游戏只有长剑一件破甲（pierce: 3）。
   - 后期敌人 armor 增长到 4-5，破甲 3 形同虚设。
   - 没有破甲 buff、技能、附魔——破甲是个一锤子买卖，没有 build 空间。

### 实施方案

**闪避（统一术语）：**
- 把闪避数值统一成单一字段：`player.statuses` 里加 `dodge_chance` 状态，stacks = 当前百分比
- 所有来源（意念甲、p_dodge、新技能）都通过 `addStatus` 累加到同一个 stacks
- battle.ts 里只在一处 roll：`if (Math.random() < dodgeChance) skip damage`
- "闪避姿态" 技能改名为 "**屏息**"，避免和闪避概念撞名
- 战斗 UI：玩家区显示"🎯 闪避 35%"芯片
- 触发动画：半屏闪烁 + 中心 "MISS" 浮字 + 敌人 0.2s 摇晃
- 新卡：
  - 道具 **烟雾弹**（rare）：本场战斗 +30% 闪避，5 回合
  - 技能 **风步**（super_rare）：下一次受击必闪避（一次性 buff）
  - 附魔 **幻影附魔**（消耗 dark 碎片 ×3）：闪避后下次攻击双倍伤害

**穿甲（build 化）：**
- 给 pierce 加 build 维度，新增：
  - 武器 **破军**（rare，奖励池）：基础伤 5，pierce 等于敌人 armor 的 50%（自适应）
  - 技能 **穿甲射**（rare）：无视全部 armor 一击
  - 附魔 **锐利**（消耗 humanoid 碎片 ×3）：所有攻击 +pierce 等于楼层数
  - 特性 **洞察**：每张 pierce +1
- 攻击 log 里破甲量直接打在伤害飘字上（如 "-15 (破甲 3)"）

### 涉及文件
- `types.ts`（新状态字段）
- `battle.ts`（统一 roll、伤害飘字）
- `cards.ts`（新卡、新附魔、新特性）
- `main.ts`（闪避概率芯片、MISS 动画）
- `style.css`（动画 CSS）

## 🅓 楼层间随机事件（待实现）

### 目标
在战斗外加一层"决策"，做出每次跑酷的差异化——肉鸽核心循环。

### 触发时机
每 2 关之间（即将进入 3/5/7… 关时），目前是铁匠铺位置。
**改为：先随机事件，再铁匠铺。** 或者把铁匠铺也并入事件池作为其中一个事件。

### 事件池（6-8 个）

| 事件 | 描述 | 玩家选择 |
|---|---|---|
| 🛒 **流浪商人** | 任选 5 张（来自奖励池）展示 | 选 1 张，消耗对应稀有度的灵魂碎片：common 3、rare 5、epic 10 |
| ⛲ **古老神龛** | 献祭 5 HP 永久换 1 张稀有牌 | 接受 / 拒绝 |
| 💀 **诡异棺木** | 50% 1 件 epic / 50% 中诅咒（永久 -2 攻击） | 开 / 不开 |
| 🔥 **远行篝火** | 回 30% HP 或升级 1 张牌至 +形态（暂时未实现牌升级，可延后） | 二选一 |
| 🎲 **赌徒** | 弃 2 张换 1 张随机奖励池牌 | 接受 / 拒绝 |
| 🐦‍⬛ **流浪术士** | 免费选 1 个特性的小弱化版（stacks=0.5 效果） | 三选一 |
| 📦 **神秘宝箱** | 抽 1 张 epic（30% 概率有陷阱：HP -30%） | 开 / 不开 |
| ⚒ **铁匠铺**（保留作为事件之一） | 当前铁匠铺逻辑 | 现有 |

### 实施方案
- 新 GamePhase: `"floor_event"`
- 新 EventDef 接口：`{ id, name, icon, desc, options: EventOption[] }`
- `event.ts` 新文件，事件池 + 触发逻辑
- 在 `afterPerkReward` 里：每 2 关后改为 → 先 floor_event → 完成后再 forge → 再 startNextFloor
- main.ts 新增 `renderFloorEvent()`：图标 + 描述 + 选项按钮
- 选择后立刻应用效果，自动进入下一阶段
- 部分事件需要"花钱"（碎片消耗），UI 要显示当前碎片库存

### 涉及文件
- `types.ts`（GamePhase 加 floor_event）
- `event.ts`（新文件，事件定义与逻辑）
- `game.ts`（流程调度）
- `main.ts`（渲染）
- `style.css`（事件 UI）

### 注意
- 新增的"+ 形态升级"和"碎片消耗买卡"是新机制，第一版可以简化（没有 + 形态时改为加 1 张 rare）
- 事件之间互斥：每两关只触发一个，避免节奏拖沓
- 第 N+1 次重复跑酷时，不要重复出现"棺木陷阱"等惩罚事件——可以引入 hash(seed) 排重（高级，可延后）

## 🌄 V2 · 3D 关卡版重构（远期，未启动）

> 下一个大版本：从纯 2D 卡牌界面 → 类纪念碑谷的 3D 关卡场景 + UI HUD 叠加。
> 由 Kuan（kuanqinc）主导场景开发，分工详见 CLAUDE.md。本节是设计上下文档案。

### 核心愿景

- **每关 = 一个独立的 3D 场景**
- 视角：等距正交 + shot-based 慢摇（已在 `prototypes/scene-f1-outside.html` 验证）
- 玩家在场景里**沿可见路径走** → 走到节点 → 触发遭遇（战斗 / 事件 / 商店 / 铁匠铺）
- **UI 模式：HUD 悬浮** —— 手牌 / HP / 状态条 / 战斗 modal 全部作为透明 HTML 叠层，3D 场景始终可见在背景

### 已就绪的资产（待清点确认）

- `prototypes/scene-f1-outside.html` — F1 塔外场景 V1（参考标准）
- `prototypes/character-gallery.html` — 主角 + NPC + 宝箱 3D 模型
- 待确认是否齐全：
  - 5 个事件 NPC（商人/赌徒/神社/巫师/宝箱）
  - 铁匠铺 NPC
  - 玩家
  - 敌人（5 种族 × 3 tier — 估计还要做 / 走拼装方案）

### 待解决的两个大系统

#### 1. 场景 ↔ 随机路线 自适应匹配

现状：`map.ts:generateFloorMap` 生成抽象节点图 `{layer, col, next[]}`，玩家点击节点切换战斗。
3D 化后需要把节点图映射到场景里的实际位置。三种方案待选：

- **A · 固定场景 + 候选位**：场景手工搭，节点预留 4-5 个候选位，按 generateFloorMap 数据挑能连通的位置点亮
- **B · 程序化场景 + 路径**：场景以 tile/region 为单位"长"出来，路径决定地形布局
- **C · 混合**：手工核心区（boss 房 / 入口 / 标志性地标）+ 中间随机段程序拼

倾向：先做 A（手工掌控视觉），数据驱动节点位置只挑/不生。等关卡数 ≥ 6 之后再考虑 B/C 的可复用性。

#### 2. 怪物身体组件随机拼装

现状：`enemies.ts:buildRandomEnemy` 用固定种族模板 + 数值随机。
3D 化后：模块化拼装（用户原话）→ 头 / 躯干 / 四肢 / 武器 / 尾 / 翅膀 等组件库。

两种数值耦合方案待选：

- **组件 = 数值 source**（构筑感强）：
  - "巨头" → baseHp +30%，攻击 hits × 2
  - "毒刺尾" → 攻击附 +N poison
  - "盾臂" → armor +2
  - 玩家看外形能预判机制 → strategic prediction
  - 实现成本高（每组件要绑机制）

- **组件 = 纯视觉**（实现简单）：
  - 拼装只决定外观，战斗数值仍按 tier × floor 表格生成
  - 视觉多样性 ≠ 战斗多样性
  - 实现快，但战略深度低

倾向：第一版先做"组件 = 纯视觉"快速跑通；后续再选若干旗舰组件（巨头/毒尾/盾臂）做数值耦合，逐步过渡到 v1。

### 架构转场点

主游戏现状（`src/main.ts`）UI 模式：
```ts
stageEl.innerHTML = "...";  // 全屏 DOM 铺满
```

3D 化后：
```html
<div id="canvas-3d"></div>          <!-- Three.js 场景占满 -->
<div id="ui-hud">                    <!-- 透明 overlay，叠在 canvas 之上 -->
  <div id="hand">...</div>           <!-- 手牌 HUD -->
  <div id="hp-bar">...</div>
  <div id="modal-layer">...</div>    <!-- 战斗 modal -->
</div>
```

CSS：3D 容器 `position: fixed; inset: 0; z-index: 0`；UI HUD `position: fixed; inset: 0; z-index: 10; pointer-events: none` + 子元素 `pointer-events: auto`。

### 视觉规范（已定稿，原型固定）

- 视角：等距正交相机 + shot-based 慢摇
- 渲染：Three.js + MeshToonMaterial + 5 段 gradientMap + UnrealBloomPass
- 调色：暗紫底 `#1f1828` + 火光暖橙 + 紫色符文 `#c868ff`
- 节点系统：每节点 `next[]`，玩家点底栏按钮逐个走过
- 雾：fog near 18 / far 60
- 移动端竖屏：相机 `d = aspect < 1 ? 9 : 12`

### 已踩坑（迁移时注意）

- **Z fighting**：贴表面装饰物抬高 ≥0.03；透明大面积 mesh 加 `depthWrite: false` + `renderOrder`
- **Three.js Clock**：`getDelta()` 每帧只调一次；取累计时间用属性 `clock.elapsedTime`，不用方法 `getElapsedTime()`（俩抢内部 oldTime）

- **本主**：`src/*.ts` 主游戏逻辑 / 战斗 / UI
